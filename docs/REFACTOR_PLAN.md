## 2. Module Contract

Every module must provide:

| File | Required | Purpose |
|---|---|---|
| `config.yaml` | Yes | Metadata, version, dependencies, iac_type, params |
| `README.md` | Yes | What it does, how to use standalone |
| `tests/` | Yes | Module-owned tests |
| `cdk/` | If iac_type=cdk | CDK construct(s) |
| `terraform/` | If iac_type=terraform | Terraform module |
| `cfn-template.yaml` | If iac_type=cdk | Pre-synthesized CFN template (auto-generated by CI) |
| `terraform/main.tf` | If iac_type=cdk | Thin HCL wrapper around cfn-template.yaml |
| `deploy.py` / `deploy.sh` | If iac_type=external or config-only | Custom deploy logic |
| `assets/` | Optional | Dashboards, templates, agent configs |

### config.yaml spec

```yaml
name: subscription
version: 1.2.0
iac_type: cdk                  # cdk | terraform | external | config-only
dependencies:
  - core/custom-resources
  - governance/permissions
params:
  - name: account_id
    required: true
  - name: edition
    required: false
    default: ENTERPRISE
artifacts:
  - cfn-template.yaml          # only for cdk modules
  - terraform/main.tf          # only for cdk modules
```

## 4. IaC Parity Strategy — Pre-Synthesized CFN Bridge

**Problem:** A Terraform-only customer shouldn't need to install the CDK CLI to use a CDK-native module.

**Solution:** Every CDK module ships a pre-synthesized `cfn-template.yaml` + a thin Terraform wrapper. The burden is on the module author (who has CDK), not the customer.

### How it works for a Terraform-only customer

```
manifest.yaml
     │
     ▼
  orchestrator.py groups by iac_type
     │
     ├── terraform modules ──► terraform apply (native)
     │
     ├── cdk modules ─────────► terraform apply
     │                          (via terraform/main.tf → cfn-template.yaml)
     │
     └── config-only ─────────► API calls / config push
     
     No CDK CLI needed.
```

### Thin Terraform wrapper (per CDK module)

```hcl
# use-case-accelerators/idp-accelerator/terraform/main.tf
variable "account_id" { type = string }
variable "region"     { type = string }

resource "aws_cloudformation_stack" "idp_accelerator" {
  name          = "idp-accelerator"
  template_body = file("${path.module}/../cfn-template.yaml")
  capabilities  = ["CAPABILITY_IAM", "CAPABILITY_NAMED_IAM"]

  parameters = {
    AccountId = var.account_id
    Region    = var.region
  }
}
```

### CI automation

Module CI pipelines run `cdk synth > cfn-template.yaml` on every merge to keep the artifact in sync.

## 5. IaC Type Tiers

| Tier | Modules | Primary IaC | CFN Bridge? | Notes |
|---|---|---|---|---|
| Tier 1 — CDK-native | Governance, data-sources, core | CDK | Yes — ship cfn-template.yaml + TF wrapper | Tightly coupled to Quick Suite CDK stack |
| Tier 2 — External | Use-case accelerators (IDP, LMA) | Varies | Yes — pre-synth if CDK, native if TF | Git subtree'd from external repos |
| Tier 3 — Config-only | AgentCore agents | None | N/A | Deployed via API calls / config push |

## 6. External Accelerator Strategy

**Now:** Git subtree — inlines external code, no submodule UX pain, `git subtree pull` to update.

**Later (3+ accelerators):** Migrate to a package manager pattern:
- Accelerators publish versioned artifacts to S3/CodeArtifact
- `bootstrap.sh` pulls them into a gitignored `vendor/` directory
- Semantic versioning instead of commit SHAs

## 7. Deploy Sequence

```bash
#!/bin/bash
# deploy.sh — single entry point

set -euo pipefail

# 1. Pull external accelerators, validate manifest
./bootstrap.sh

# 2. Resolve dependencies, validate params
python3 core/utils/orchestrator.py validate

# 3. Deploy CDK modules (if customer uses CDK)
python3 core/utils/orchestrator.py generate-cdk
cd templates/customer-project/cdk-app && cdk deploy --all && cd -

# 4. Deploy Terraform modules (if any in manifest)
python3 core/utils/orchestrator.py generate-terraform
cd templates/customer-project/tf-app && terraform init && terraform apply -auto-approve && cd -

# 5. Deploy external modules with custom deploy scripts
python3 core/utils/orchestrator.py deploy-external

# 6. Push config-only modules (agent configs, API calls)
python3 core/utils/orchestrator.py deploy-config
```

## 8. Migration Map (Current → New)

| Current Location | New Location | Notes |
|---|---|---|
| `lib/amazon-quick-suite-starter-kit-stack.ts` | Split across `governance/*/cdk/`, `data-sources/*/cdk/`, `core/constructs/` | Break monolith stack into module constructs |
| `lib/app.ts` | `templates/customer-project/cdk-app/bin/app.ts` | Becomes manifest-driven |
| `lambdas/src/` | Split into relevant modules or `core/custom-resources/` | Based on what each Lambda does |
| `operator_tools/` | `core/utils/` | Operator CLI stays centralized |
| `docs/` | `docs/` | Stays at root, update content |

## 9. Decisions Log

| Decision | Choice | Rationale |
|---|---|---|
| Module discovery | Manifest file (`manifest.yaml`) | Simple, no tooling overhead |
| Versioning | Per-module (`config.yaml` → `version`) | Independent release cycles |
| External accelerators | Git subtree now, package manager later | Pragmatic start, scales later |
| Testing | Each module owns `tests/` | Module independence |
| IaC parity | Not required — CFN bridge for CDK modules | Don't force TF shops to install CDK |
| Custom resources | Centralized in `core/custom-resources/` | Avoid duplication across modules |
